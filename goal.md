## 0. 名字

产品名:**线条小马**。

---

## 1. 一句话目标

一个**极轻的 Web app**:一群人聚会喝酒时,围着**一台手机**,看几只可爱的二维线条小马赛跑,过程中抢道具搞事情,最后输的人喝酒。点开链接即玩,无需安装、无需注册、无需联网对战。

---

## 2. 产品哲学(动手前必读)

这些不是装饰,是决定每一个实现取舍的根本原则。

- **它是聚会工具,不是游戏。** 用户打开它只为一件事:快速、热闹地决定"谁喝"。决定完就关掉。**不要做留存系统、不要做日活、不要做养成进度、不要做账号体系。** 任何让产品变"重"的东西都违背初衷。
- **单机单屏是刻意的设计,不是偷懒。** 所有人盯着同一块屏幕等揭晓——那个**集体围观的瞬间**才是产品的全部价值。不做扫码、不做多设备同步、不做联机。一台手机大家传着看、一起喊"快抓那个道具",这才像聚会。
- **价值压缩在两个高光时刻:抢道具瞬间 + 揭晓输家瞬间。** 道具划过屏幕大家伸手去点、道具生效时的夸张动画、最后谁输了揭晓的那一下——这几个瞬间那一桌人的反应(笑、起哄、惨叫)就是这个产品存在的理由。**把大头打磨精力放在这些瞬间的动画和音效上。** 其它界面能简陋就简陋。
- **动画就是"味道"本身。** 这个产品不是靠功能取胜,是靠那几个手绘逐帧动画的神韵——小马摔倒的样子、大哭的样子、嘲讽的样子。这是用户愿意打开它而不是别的 app 的核心原因。所以美术不是装饰层,是产品本体。
- **传播靠口碑复述,不靠买量。** 这一桌玩过的人,会在下一桌主动说"我有个 app 巨好玩"。所以产品要有**能被一句话描述、能截图发群的记忆点**(某只魔性的小马、某个被导弹炸飞的桥段)。门槛低(发链接秒进)+ 记忆点强 + 高光瞬间反应值拉满,这三件事叠起来,才能让下次喝酒大家点开的是这个,而不是别的。
- **轻到一个人零成本维护。** 纯静态前端,零后端,所有美术资产预先生成好打包进去,运行时不调任何 API。收入低没关系,维护成本接近零,这才可持续。

## 3. 美术工作流:Gemini 预生成(必读,这是本项目的特色)

### 核心原则:资产开发期生成,运行时纯播放
- **所有视觉元素(地图/赛道、小马角色、道具、UI)都用 Gemini API 在开发阶段预先生成**,切好、打包进 app 作为静态资源。
- **运行时(用户玩的时候)app 不调用任何 Gemini / 任何 API**——纯静态加载本地资产播放。原因:运行时实时生成会引入延迟、API key 暴露、按调用付费、网络依赖,全部违背"轻、零后端、点链接秒玩、零成本维护"的定位;而且酒桌高光瞬间要的是即时、丝滑、可预测的动画,实时生成做不到。
- Gemini 在这里是**开发期的美术生产工具**,不是运行时依赖。

### 风格统一:以用户的概念图为准
- 用户会提供**一张概念图**,确立"线条小马"的画风(二维、韩国线条小马/线条小狗那种极简手绘矢量感)。
- **用 Gemini 生成所有视觉元素时,都必须以这张概念图为风格参考**,保证地图、角色、道具、UI 风格统一。生成时把概念图作为参考图输入,并在 prompt 里锁定线条粗细、配色、留白、笔触等风格特征。

### 角色动画 = 序列帧(sprite sheet),不是矢量动画
**这是和上一版的根本区别,务必按此实现:**
- 角色的**身体动作动画**(跑步、冲刺、摔倒、大哭、嘲讽等)**必须用逐帧序列帧播放**。用 Gemini 一次生成同一个动作的**多帧连续画面**,切帧后拼成 sprite sheet,运行时按帧率循环或单次播放。这种逐帧手绘的播放才有"味道",这是产品的灵魂,**没有商量余地**。
- **不要用 Rive、不要用 Lottie、不要用矢量状态机**——上一版的 Rive 方案作废,改为序列帧。

### 动画 vs. 非动画的明确边界
- **位置移动**(小马在赛道上从 A 点平移到 B 点):**不需要逐帧动画**,用代码补间(tween)平滑移动 sprite 的坐标即可。省素材、省力。
- **身体动作**(跑步、冲刺、摔倒、大哭、嘲讽,以及道具生效的表演):**必须逐帧序列帧播放**。
- 一句话:**"挪位置"用 tween,"做动作"用序列帧。**

### 需要生成的角色动作帧(每只小马一套,或一套共用换色)
至少包含这些状态的序列帧:
- `idle` 待机(开局/等待,轻微呼吸/摆动,可少量帧循环)
- `run` 跑步(核心循环动画,赛跑全程播放)—— **两条腿直立奔跑(拟人姿态)**,见下方姿态设定
- `sprint` 冲刺(吃到加速道具时的夸张快跑)—— **四脚着地的兽态狂奔**,见下方姿态设定
- `trip` 摔倒(被香蕉绊倒等,夸张地摔)
- `cry` 大哭(输了 / 被惩罚时,夸张哭戏)
- `taunt` 嘲讽(赢了 / 领先时嘚瑟)
- (按道具需要可加:`hit` 被导弹炸到、`ride` 搭车前进等)

### ⭐ 关键姿态设定:两脚跑 vs 四脚冲刺(反差萌,务必体现)
- **正常跑步(`run`):小马用两条后腿直立奔跑**,像拟人化的卡通角色那样,这是为了"可爱"。这是赛跑全程的默认姿态。
- **冲刺(`sprint`):吃到加速道具时,小马切换成四脚着地的真·马姿态狂奔**,爆发力拉满。
- 这个**从"两脚萌跑"到"四脚兽奔"的姿态切换本身就是一个反差萌的看点**,是产品记忆点的一部分。生成 `run` 和 `sprint` 两套序列帧时,必须明确区分这两种姿态(不是同一姿态的快慢版本),并且过渡要有戏剧性。
- 实现上:`run` 和 `sprint` 是两套独立的序列帧,触发加速道具时从 `run` 切到 `sprint` 播放,道具效果结束后切回 `run`。

### 资产组织建议
- 角色帧:可先做**一套基础小马 + 多种配色**(色相替换或分别生成),用颜色区分玩家。
- 把每个动作的帧数、帧率、是循环还是单次播放,做成**配置文件(如 JSON)**,渲染层读配置播放,方便之后调整/加动作不改代码。

## 4. 范围(Scope)

### 做什么(本次就做全)
- 单页 Web app,移动端浏览器优先(竖屏,手机上玩)。
- 开局界面:设置参赛小马数量(建议 3–6 只),可给小马起名/选颜色(可选,别做复杂)。
- 一条赛道,几只小马赛跑,跑步用序列帧、移动用 tween。
- **道具系统(本次核心,见第 5 节):** 道具像流星一样划过屏幕,用户点击抓取,触发对应的夸张序列帧动画和效果。
- **戏剧性的赛跑过程 + 揭晓瞬间**:有逆转、有意外、有反应。
- 结算:明确指出**输家**(最后一名喝酒),揭晓动画要有张力。
- "再来一局"按钮一键重开。

### 明确不做(排除项,别自作主张加上)
- ❌ 扫码 / 多设备 / 联机同步(就是一台手机大家看)
- ❌ 账号 / 登录 / 注册
- ❌ 养成 / 进度 / 段位 / 收集系统
- ❌ 双端原生 app(只做 Web)
- ❌ 运行时调用 Gemini 或任何 API(资产全部预生成打包)
- ❌ Rive / Lottie / 矢量状态机动画(改用序列帧)
- ❌ 复杂的商店 / 订阅体系

## 5. 道具系统(本次核心,详细实现)

### 掉落与抓取机制
- 道具在赛跑过程中**随机出现,像流星/划过屏幕一样飞过**,用户用手指点击抓取。
- **稀有度决定飞行速度和效果强度:越罕见 → 飞得越快 → 越难点中 → 效果越强大。** 常见道具飞得慢、好抓、效果温和;罕见道具飞得快、转瞬即逝、抓住了效果炸裂。这个"越强越难抓"的张力就是道具系统的乐趣核心,也是酒桌上大家一起伸手抢、喊叫的来源。
- 点中道具后,播放**对应的生效过程动画**(序列帧表演 + 必要的场景特效),要夸张、要有反应、要让旁边人看到结果。

### 道具效果清单

**常见(慢、易抓、效果温和):**
| 道具 | 效果 | 动画表现 |
|---|---|---|
| 奋力冲刺 | X 号小马向前加速一段 | X 号播放 `sprint` 冲刺序列帧 + 速度线特效 |
| 香蕉皮 | X 号小马被绊倒,后退/停顿 | X 号播放 `trip` 摔倒序列帧 + 香蕉皮道具 |
| 惩罚 +1 | 目标的"喝酒惩罚"数 +1 | 惩罚数字跳动 +1 的反馈动画 |
| 惩罚 -1 | 目标的"喝酒惩罚"数 -1 | 惩罚数字跳动 -1 的反馈动画 |

**罕见(快、难抓、效果炸裂):**
| 道具 | 效果 | 动画表现 |
|---|---|---|
| 乾坤大挪移 | X、Y 号小马**位置互换** | 两只小马交换位置的夸张演出(可配特效) |
| 导弹空袭 | **当前第一名**被导弹炸到,**直接后移一大段** | 导弹飞入 + 爆炸特效 + 第一名播放 `hit`/被炸飞 + 后退 |
| 搭便车 | **当前最后一名**搭车,**往前挪一大段** | 最后一名搭上车辆往前冲的演出 |

> 实现要点:道具效果作用于**逻辑层**(改名次/位置/惩罚数),动画在**渲染层**表演逻辑结果。两者解耦(见第 6 节),这样以后加新道具就是"加一条效果 + 一套帧 + 一段配置",不动核心。

### 惩罚(喝酒量)概念
- 每只小马身上有个"惩罚值"(= 最后要喝的量),道具可以 +1/-1。
- 结算时:最后一名是输家,结合其惩罚值得出最终喝酒量。规则做简单清晰即可,别过度设计。

## 6. 核心架构:逻辑层与渲染层解耦

- **逻辑层(纯逻辑,无渲染):** 用随机种子驱动,负责每只小马每帧的速度/位置、名次变化、道具的掉落时机与稀有度、道具效果的结算(加速/绊倒/换位/炸退/搭车/惩罚增减)、最终输家判定。逻辑层应可独立运行和测试。

### ⭐ 输赢三层影响模型(逻辑层核心规格,务必按权重实现)
最终名次由**三个层级叠加**决定,三者对结果的影响力大致按下列权重分配。**这是产品手感的命门,不要自行更改权重或省略任何一层。**

| 层级 | 占比 | 性质 | 说明 |
|---|---|---|---|
| ① 基础奔跑速度 | **40%** | 纯运气,**开局前就定好** | 每只小马在比赛开始前随机分配一个基础速度,纯随机、不可预测。这是"基本盘"。 |
| ② 随机减速事件 | **30%** | **非道具**的过程随机事件 | 比赛过程中随机触发的负向事件:自己摔倒、停下来喘气等。作为对前进的负向扰动。**用 B 方案(追赶机制),见下。** |
| ③ 道具效果 | **30%** | 用户主动操作 | 用户抓取并使用道具带来的影响(加速/绊倒/换位/导弹/搭车/惩罚增减)。 |

**关键设计含义:** 基础速度占四成是基本盘,**但锁不死结果**——减速事件(30%)+ 道具(30%)合计 60% 的影响力大于基础速度(40%)。所以**手气好的小马大概率领先,但完全可能被连串倒霉事件拖垮、或被对手道具翻盘**。这正是"有戏剧性、能逆转、还想再来一局"的来源。务必让这三层的相对影响力在实际跑动中体现出这个比例,而不是基础速度一家独大。

**减速事件采用 B 方案(追赶机制 / 橡皮筋):**
- **当前领先的小马,触发减速事件的概率略高**(像马里奥赛车的 rubber-band AI),制造"领先者总被拖回来"的拉锯感,让悬念维持到最后、揭晓更晚、更热闹。
- ⚠️ **但要克制,别让人看出来在"作弊":** 追赶倾向要温和(领先者概率略升、落后者略降),保留足够随机性,绝不能变成"第一名必被惩罚、最后一名必被照顾"的硬规则。目标是"领先者没那么稳",不是"领先毫无意义"。把领先位次到减速概率的映射做成**可调参数**,方便后续手感微调。
- 减速事件触发时,对应小马播放 `trip`(摔倒)或一个"喘气/停顿"的序列帧表演,让观众看到它为什么慢下来。

> 实现建议:把每帧的"有效前进量"理解为 `基础速度分量(40%权重) − 减速事件分量(30%权重) + 道具分量(30%权重)` 的加权合成。具体数值化方式开发者可定,但三者对最终名次方差的贡献要大致吻合 40/30/30,且模型可独立测试验证(例如多次模拟统计各层对结果的影响占比)。

- **渲染层(PixiJS):** 只负责把逻辑结果"演"出来——sprite 的 tween 移动、序列帧动作播放、道具划过屏幕、点击命中检测、特效、镜头节奏、UI。
- 好处:换皮肤、加小马、加道具、调数值,大多只动数据/配置,不动渲染核心。**道具效果通过"事件"注入逻辑层**,渲染层订阅事件播放对应动画。

### 赛跑手感
- **随机但有戏剧性,由上面的三层影响模型驱动(40/30/30)。** 不要让结果一开始锁定。基础速度是基本盘,减速事件(追赶机制)持续把领先者拖回来,道具制造突发翻盘——三者叠加让过程中名次频繁变化、领先者被反超、黑马逆袭。意外 = 笑点 = 还想再来一局。
- 结果对所有小马公平随机(没人能预测谁赢),但**演出要有张力**,别让某只一路平稳领先到底。

### 揭晓瞬间(灵魂,重点打磨)
- 临近终点放慢节奏、制造悬念(可慢镜、可镜头拉近)。
- 输家确定那一刻,夸张反差动画:输的小马播放 `cry` 大哭/瘫倒,赢的播放 `taunt` 嘲讽。
- **音效必做:** 酒桌很吵,洗脑赛跑 BGM + 输了时夸张的惨叫/喇叭音效 + 抓到罕见道具的爽快音效,是比视觉更强的记忆和传播抓手。
- 这些瞬间反复调:帧率、节奏、缓动、音画同步,打磨到"那一桌人会笑出声"。

## 7. 技术栈(已确定)

| 层 | 选择 | 理由 |
|---|---|---|
| 美术生成 | **Gemini API**(开发期) | 预生成地图/角色帧/道具/UI,以用户概念图为风格参考;运行时不调用 |
| 构建 | **Vite** | 快、零配置、Web 部署友好 |
| 渲染 | **PixiJS** (WebGL 2D) | sprite 序列帧播放、tween 移动、粒子特效、点击命中检测都强;比 Phaser 轻(渲染器而非完整引擎);比纯 DOM/CSS 可控,高光瞬间需要精确编排时间轴和缓动 |
| 角色动画 | **序列帧 / sprite sheet**(PixiJS `AnimatedSprite`) | Gemini 生成多帧 → 切帧 → 按配置播放;逐帧手绘才有"味道" |
| UI 外壳 | **原生 JS** 或 **Svelte** | UI 极简,**不要用 React**;想要框架便利就用 Svelte(产物小),否则原生 JS + PixiJS 足够 |
| 部署 | **Cloudflare Pages** 或 **Vercel** | 纯静态,白嫖额度足够,发出去就是一个链接 |

### 避免被带偏
- **不要 Rive / Lottie / 矢量状态机** —— 已改为序列帧。
- **不要运行时调 API** —— 资产全部预生成打包。
- **不要 Unity** —— 二维轻量小游戏的杀鸡用牛刀,WebGL 加载慢、包体大。
- **不要 React** —— UI 太简单,框架是负担。

## 8. 变现(轻量,别做重)

顺着"收入低成本低、一个人维护"的定位:
- **免费玩。** 每玩 X 局(比如 3–5 局)插一个简短广告。
- **去广告 = 一次性买断**(几块钱),不做订阅、不做会员体系。
- **不做** cosmetics 商店、不做激励视频解锁、不做复杂 IAP。
- 广告/分析 SDK 预留接口即可,**第一版可先不真正接广告**,先把核心体验跑通。

## 9. 落地顺序(实施路径)

1. **美术先行(关键路径):** 用 Gemini 按用户概念图生成第一批资产——一只小马的 `idle/run/trip/cry/taunt` 几套动作帧、赛道背景、几个核心道具图标、必要 UI。切帧、建 sprite sheet、写帧配置 JSON。先用**一只小马 + 几个动作**验证画风和"味道"对不对,再批量铺开和加配色。
2. **搭骨架:** Vite + PixiJS,移动端竖屏页面,一条赛道。
3. **小马赛跑机制:** 逻辑层(随机种子→速度/名次→输家)+ 渲染层(tween 移动 + `run` 序列帧循环)。先让"一局能跑完、能跑出输家"。
4. **道具系统:** 道具流星式划过 + 点击命中检测 + 稀有度决定速度/强度;接入全部常见/罕见效果及其生效动画;道具效果以事件注入逻辑层、渲染层订阅播放。
5. **打磨高光瞬间(大头精力):** 抓道具的爽感、道具生效的夸张演出、临近终点的悬念、输家揭晓的反差动画;配齐 BGM + 惨叫 + 抓取音效。
6. **开局界面 + 再来一局:** 设置小马数量、可选起名/选色;一键重开。
7. **部署成链接:** Cloudflare Pages / Vercel,纯静态,发出去即玩。

## 10. 验收标准

- 打开一个链接,手机竖屏,无需任何登录/安装,直接能开始一局。
- 3–6 只线条小马赛跑:移动用 tween,跑步/冲刺/摔倒/大哭/嘲讽**用序列帧播放**,味道到位。**正常跑步是两脚直立萌跑,吃到加速道具冲刺时切换成四脚兽态狂奔**,反差到位。
- 道具像流星划过屏幕,可点击抓取;**越罕见飞得越快越难抓、效果越强**;抓到后播放对应的夸张生效动画。常见与罕见道具效果全部实现(冲刺/香蕉/惩罚±1;换位/导弹炸退第一名/搭车带最后一名)。
- 过程有名次逆转和戏剧性,结果公平随机。**输赢由三层影响模型驱动:基础速度 40%(开局定、纯运气)、随机减速事件 30%(非道具,领先者略易触发的追赶机制)、道具 30%;三层相对影响力大致吻合该比例(可通过多次模拟统计验证)。**
- 揭晓输家时有**悬念 + 夸张反差动画 + 音效**,让人想给旁边人看、想再来一局。
- 所有视觉资产为 Gemini 预生成、风格统一(以概念图为准)、本地打包;**运行时零 API 调用**。
- "再来一局"一键重开;纯静态部署,一个链接可分享,点开即玩。
- 代码结构:逻辑层与渲染层解耦;序列帧动作走配置驱动;道具效果通过事件注入,易扩展。
